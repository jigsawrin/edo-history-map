# 歴史年表

## 目的と対象

歴史年表は、既存の滋賀・戦国と京都・幕末の地点・歴史テーマを、監査済みの年代順でたどる入口である。
初版は35項目（滋賀・戦国17、京都・幕末18）、地点関係42、テーマ関係62、関連テーマ21を収録する。
東京・江戸8,788地名は出来事との構造化関係がないため、自動登録しない。

滋賀・戦国は1560年頃から1583年、京都・幕末は1858年から1868年の採用項目を持つ。
戦国期と幕末期の間には大きな空白がある。本年表は1467～1868年の日本史を連続的・網羅的に
説明するものではなく、本アプリ収録地点から固定出典で直接確認できる出来事だけを掲載する。

## 独立キュレーション

正本は`data-curation/historical-timeline.json`、採否記録は`audit/historical-timeline-review.md`である。
既存GeoJSON、地点ID、地点説明、テーマID、テーマ関係へ日付を混在させない。ブラウザ実行時の追加fetchはなく、
TypeScriptの静的importで取り込む。任意URLは保持せず、出典は京都・滋賀の固定出典IDから解決する。

各項目は一意なASCII ID、一意な整数`order`、track、type、独自要約、1件以上の関連地点、1件以上の
関連テーマを持つ。地点のsource IDは、その地点自身のsourceIdsに含まれるものだけを登録する。

## 日付モデル

日付は`displayJa`、明示的な開始・終了の年・月・日、`precision`、`calendarBasis`、任意の注意で表す。
precisionは`day`、`month`、`year`、`range`、`circa`、calendarBasisは`japanese-lunisolar`、
`gregorian`、`year-only`、`mixed`に限定する。`order`は人が監査した年代順で、開始年の逆行を拒否する。

次を行わない。

- `dateDisplayJa`、`periodDisplayJa`、要約、地点名、テーマ名、出典本文から年月日を自動抽出する。
- 月不明を1月、日不明を1日、`YYYY-00-00`として扱う。
- 不完全日付をJavaScriptの`Date`へ変換する。
- 旧暦日を換算根拠なしにグレゴリオ暦へ変換する。

旧暦日は当時の日本の暦による表示であることを明示する。年まで確認できる項目は月日を補わず、
期間は単日の出来事に見せず、circaは正確な日付に見せない。静的HTMLの`datetime`は、年精度なら西暦年、
正確なグレゴリオ日だけ完全日付を使い、旧暦の月日を機械可読なグレゴリオ日として出力しない。

## 採用原則

既存地点、既存テーマ、日付根拠、地点との関係説明、地点所属出典をすべて確認できる候補だけを採用する。
後世の碑・墓所・再建建物は当時の出来事地点や建物そのものと断定せず、広い戦場・寺域・城域は代表地点と
出来事範囲を区別する。不採用候補も監査表へ残し、弱い根拠で件数を水増ししない。

## 地図版UI

ツールバーの「歴史年表」は通常HTMLパネルを開く。時代区分、出来事種別、100文字上限の検索、年代順一覧、
詳細、関連地点、関連テーマ、固定出典、静的年表リンクを提供する。検索は既存のNFKC・かな・空白・小文字・
制御文字除去を再利用し、タイトル完全一致、別名完全一致、タイトル前方、タイトル部分、別名部分、日付、要約の
順で順位付けする。入力からRegExpを作らない。

テーマと年表の関連地点選択は共通関数を使う。固定dataset IDから地域・歴史年代を決め、必要なら地域と年代を
切り替え、検索モデルキャッシュから地点を取得し、世代番号で古い非同期結果を拒否する。その後、panと既存情報
カードを使い、カード見出しへフォーカスする。モバイルはパネルを閉じ、カード閉鎖時は対応する開くボタンへ戻る。

年表の検索、絞り込み、選択はURL、hash、Cookie、localStorage、sessionStorage、IndexedDBへ保存しない。

## 静的年表と逆リンク

`npm run build:static-timeline`は次を生成する。

- `/timeline/`
- `/timeline/shiga-sengoku/`
- `/timeline/kyoto-bakumatsu/`

全項目は時代別ページの`article`と一意アンカーで表示し、関連テーマ、地点アンカー、関係説明、固定出典、地図版、
前後項目へ通常リンクで移動できる。京都・滋賀地点ページは`relatedPlaces`、個別テーマページは
`relatedThemeIds`から逆引きする。名称一致を使わない。

静的ページはJavaScript、form、外部画像、外部フォント、runtime fetch、生成時刻を含まない。厳格CSP、
HTML・属性エスケープ、HTTPS固定origin、認証情報なし、`noopener noreferrer`、重複ID・空リンク拒否、
決定的出力とSHA監査を行う。manifest schema 3へ件数・内訳・関係数・HTML/CSS/入力SHA・逆リンク数を記録する。

## 確認範囲と制約

自動テストはレジストリ、検索、UI、静的生成、リンク、manifest、既存地点・テーマ回帰を対象にする。
外部ChromeではPC、キーボード、390×844、JavaScript無効、200％・400％、印刷、consoleを確認する。
スクリーンリーダーの実読み上げと物理モバイル実機は、実施した場合だけ確認済みと報告する。

この初版は収録地点に結び付く二時代だけを扱う。次候補を追加する場合も、既存地点・テーマ・固定出典、
日付精度、代表地点の限界を個別監査し、空白を自動生成や推測で埋めない。

## 機械可読日付とGregorian検証

静的HTMLは`precision: year`だけを年の`datetime`として出力する。`circa`と`range`は通常の`span`、
`month`と`day`は`calendarBasis: gregorian`の場合だけ`time`とし、日本旧暦・mixedへ推測した
Gregorian日付を付けない。この規則は時代別ページと地点・テーマ逆リンクで共通化する。

Gregorianかつdayだけ、整数演算の閏年規則（4で割り切れ、100で割り切れる年は除外し、400で
割り切れる年は含む）と月別日数で実在日を追加検証する。一般の月1–12・日1–31検証は維持し、
日本旧暦・mixedにはGregorianの月日数を強制しない。`Date`やタイムゾーン依存変換は使わない。
